Tweet Positive HN Posts Without Coding

This posts reports on a little experiment with Node-RED, using its visual data flow editor to fetch HN posts, filter those with positive words in the comments, and tweet them on @positiveHN.

The flow

I recently came across Node-RED and wondered how good it would be as a tool to automate a few things I find more and more annoying to do manually. A friend of mine challenged me to use it to tweet “positive” posts from Hacker News (mostly because there was a sentiment analysis node as part of the built-in set of node). Extra points for making it without code (even though there is a function node to write arbitrary javascript to process messages).

Here is the result:

It works as follows:

  • Parse HN's RSS feed creates a new message (Node RED’s unit of processing) every time a new article appears in HN’s RSS feed.
  • limit 5 msg/s will make sure that no more that no more than 5 messages per second go through the rest of the flow. HackerNews seems to dislike too frequent requests, as made in the next node.
  • fetch comments requests the comment web page of the post
  • keep text only extracts the comments from the web page
  • join merges all the comments into one text
  • sentiment analyses the text
  • positive only lets only messages with a positive score through
  • title > 120 chars ? switches messages depending on whether the title of hte post is longer than 120 characters
  • truncate title shortens the long title
  • long title produces a tweet with the shortened title (adding suspension mark) and the url of the post
  • short title simply produces a tweet with the title and the url of the post
  • Tweet publishes the tweet on @positiveHN.

Persistence

Node-RED is event-based, which makes it not naturally persistence-friendly. In this example, I tried to keep track of which (positive) posts had already been tweeted, in order to avoid duplicate on Twitter. Turns out, Twitter already reject duplicates, but before I realized that, I had found two hacks to tilt at that windmill.

The first one was a really dirty hack, makeing use of the fact that an exception was thrown when trying to open an non-existent file, and that a special node was able to catch exceptions. Thus, for every message coming frome the feed parsing node, the url of the post was base64 encoded and used as a filename to open. If the open succeeded, nothing more happened, but if it failed, it meant that the url hadn’t been handled yet. The exception catching node would then initiate another flow that would tweet and save an empty file with the base64 encoded url as name. I did warn it was dirty.

A slightly-less-hacky-but-still-not-ideal working alternative was to store the urls in a file. The flow would read the file for each message, move its content somewhere in the message, and check whether the url was in the content. If so, it meant that the url had already been processed, otherwhise, the flow would go one by tweeting the url and saving it into the file. I’m not sure how parallel read and write are handled… so using a database would sound like a better idea. But again, all this was unnessary, since Twitter already does (a better) job. Yay!

Shortcomings

A better use of the persistence would be to reprocess the list of posts until they reach a positive score or disappear from the homepage. As it is, the sentiment analysis will be performed as soon as a post lands in the stream, i.e. as soon as it reaches the homepage. If not enough comments have been written by then, or too few are positive, the url will never be tweeted, even if the comments become positive afterwards.

Another issue that could be addressed by keeping track of which url has been processed is linked to the fact that the feed parsing node forgets which urls have already been sent through the pipeline between restarts of the server. Again, it is not a problem in this case because Twitter will duplicate updates, but it would be for example if we send an email instead of a tweet.
